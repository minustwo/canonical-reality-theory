Release Policy

Purpose

This document defines how material may enter the public repository.

Release principle

Public release is a separate act from private research progress.

A result may exist privately without being public-safe.

Release gates

Before release, each file must pass:

1. English-only check
2. Private-source leakage check
3. Claim-status check
4. Theorem-promotion check
5. Proof-compression check
6. Architecture fit check
7. Human approval

Public-safe material

The following are usually public-safe:

* Repository documentation
* Architecture diagrams without theorem promotion
* Status policy
* Review policy
* AI usage disclosure
* Release process documentation

High-risk material

The following require explicit review before public release:

* Theorem statements
* Proofs
* Bridge claims
* Internal audit reports
* Raw research notes
* Private repo extracts
* AI-generated mathematical reconstructions
* Drafts containing mixed-language material

Versioning

Technical releases should include:

* Version identifier
* Date
* Status label
* Scope
* Change summary
* Known limitations

No automatic publication

No private file should be copied into this repository automatically.

All public imports require explicit human approval.
