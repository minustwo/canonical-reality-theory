Methodology

Purpose

This document defines the public research methodology for this repository.

Core methodology

The MST–CRT–IET stack is developed through layered formalization:

1. Define the structural object.
2. Separate primitive concepts from derived concepts.
3. Identify admissibility conditions.
4. Distinguish uniqueness, determinization, selection, and canonicalization.
5. Track failure modes.
6. Audit bridge claims separately.
7. Preserve conditionality unless proof status changes through review.

Anti-overclaim rule

The repository must avoid the following:

* Treating architectural alignment as proof
* Treating examples as theorems
* Treating bridge intuition as equivalence
* Treating AI-generated review as peer review
* Treating private draft consensus as public validation

Public documentation method

Public documents may explain:

* Repository organization
* Research status
* Layer relationships
* Review workflow
* Release policy

Public documents must not:

* Rewrite theorem content
* Compress proofs into informal claims
* Promote unfinished results
* Import private material without approval

Audit-first development

Major claims should pass through audit before public technical release.

Audit should check:

* Definitions
* Dependencies
* Claim scope
* Directionality
* Minimality
* Counterexample exposure
* Conditional assumptions
* Public suitability
