# Legacy Technical Archive
## Purpose
This document explains how to read legacy technical material in the public CRT research repository.
The repository now has a clean public research surface, but it also preserves older deep technical archive material.
This legacy material is retained for provenance, continuity, and research traceability.
It should not be interpreted as the current top-level public architecture unless explicitly stated.
## Legacy archive examples
Legacy technical archive material may include:
- `THEORY_INDEX.md`
- `PUBLICATION_STATUS.md`
- `STATUS_GLOSSARY.md`
- `papers/foundations/`
- `papers/open-systems/`
- `proofs/supplement/`
- older scaffold files under `papers/`
These files may contain theorem statements, proof sketches, technical reports, status pages, or older terminology.
## How to read legacy material
Read legacy material through the current repository status system:
- `docs/STATUS_LABELS.md`
- `docs/CLAIMS_LEDGER.md`
- `proofs/STATUS.md`
- `proofs/supplement/STATUS.md`
- `docs/IET_NAMING.md`
A legacy artifact being public does not imply:
- peer review
- current theorem status
- global synthesis
- unconditional validity
- current terminology
- final architecture
## THEORY_INDEX.md
`THEORY_INDEX.md` is a deep technical index and legacy CRT archive map.
It is useful for understanding historical structure and technical dependencies.
It is not the current public entry page.
The current public entry page is:
- `README.md`
The current architecture reference is:
- `docs/ARCHITECTURE.md`
The current claim-status reference is:
- `docs/CLAIMS_LEDGER.md`
## PUBLICATION_STATUS.md
`PUBLICATION_STATUS.md` records publication, preprint, archival, and technical-report status.
It should not be read as a peer-review certification.
Publication status and claim status are separate.
## STATUS_GLOSSARY.md
`STATUS_GLOSSARY.md` preserves older CRT archive terminology.
Canonical status labels are now defined in:
- `docs/STATUS_LABELS.md`
If terminology conflicts, use `docs/STATUS_LABELS.md` as the canonical status-label reference.
## Proof supplements
Proof supplement files should be read through:
- `proofs/STATUS.md`
- `proofs/supplement/STATUS.md`
Do not cite proof supplements as externally peer-reviewed publications unless explicitly stated.
## Migration policy
Legacy material should be migrated gradually.
Migration should:
- preserve provenance
- avoid theorem rewrites
- avoid proof compression
- avoid claim promotion
- add status metadata before restructuring
- update AI-agent memory when architectural meaning changes
## Non-claims
This archive policy does not claim:
- all legacy materials are current
- all legacy materials are superseded
- all legacy materials are proved
- full MST–CRT–IET synthesis
- external peer review
